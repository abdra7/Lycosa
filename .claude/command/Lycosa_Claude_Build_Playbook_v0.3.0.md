# Lycosa Claude Build Playbook: Production Release Audit (v0.3.0)

This playbook outlines the prompts and steps required to execute the Production Release Audit for `v0.3.0`, ensuring the platform is hardened, fully documented, and ready for public distribution.

---

## 📋 Release Objectives
* Execute a full repository and configuration audit.
* Run security penetration tests covering dependency checks and injection vulnerabilities.
* Assess production deployment scaffolding (Docker Compose, Health Checks, SSL).
* Document release notes, changelog extensions, and readiness scorecards.

---

## 🛠️ Step 1 — Production Scaffolding Discovery

### discovery_prompt
> **Production Scaffolding Audit**
> Inspect the repository configurations in `/infra/docker-compose.yml`, `.env.example`, and `/backend/app/core/config.py`.
>
> Identify:
> 1. Secrets handling (e.g. database password encryption, JWT hash validation).
> 2. Health check interval and timeout policies across Docker containers.
> 3. Production logging configurations (JSON formats, log rotation, log levels).
>
> Verify the config and present a list of production gaps.

---

## 🔒 Step 2 — Hardening & Security Audit

### security_hardening_prompt
> **Security Penetration Probe**
> Analyze API gateway boundaries for OWASP-style vulnerabilities:
> 1. Check `/api/v1/auth` endpoints for brute-force vectors and rate-limit effectiveness.
> 2. Probe document file pickers and parsing paths for Path Traversal or Zip Slip attacks.
> 3. Verify that outbound LLM models pulled by nodes do not bypass firewall policies.
>
> List any identified vulnerabilities and suggest immediate remediation steps.

---

## ⚡ Step 3 — Load & Performance Stress Tests

### stress_tests_prompt
> **High Concurrency Performance Testing**
> Run a load simulation suite mimicking production traffic on task orchestration endpoints:
> * Simulate 100, 500, 1000, and 5000 concurrent client operations.
> * Measure database connection pool utilization (PostgreSQL session counts).
> * Monitor memory footprint and thread starvation rates.
>
> Generate a performance latency curve mapping requests to execution completion speed.

---

## 📄 Step 4 — Documentation & Release Notes

### release_notes_prompt
> **Changelog and Release Notes Compiler**
> Compile the final list of features, bug fixes, and security patches introduced in this sprint.
>
> Format updates matching the Keep a Changelog standard:
> * **Added:** Persisted dark/light theme, admin node deletion API.
> * **Fixed:** Localhost LAN IP resolution, mDNS scanner Windows error skips.
> * **Security:** Hardened token headers and rate-limiting limits.
>
> Update the main CHANGELOG.md file and prepare the final Release Report.

# Additional Improvements

## Zero-Configuration Installation
- Remove mandatory .env for end users.
- Auto-generate secrets on first run.
- Auto-create config/database/directories.
- Safe defaults built into application.
- Clone -> Install -> Run.

## Theme Requirements
- Default theme MUST be Light Mode.
- User can switch to Dark Mode.
- Persist theme locally.
- Restore saved theme on every launch.
- Prevent flash of wrong theme.

## Acceptance Criteria
- Fresh clone works without .env
- Docker works immediately
- Local execution works immediately
- No manual configuration
- Theme persistence verified
- Production-ready release only

# 📁 Claude Documentation Policy

## Claude Build Documentation

All Claude-specific build and development documentation is tracked in Git and hosted on GitHub, but it MUST be organized and kept within a single folder called `.claude/command/`.

Examples of files inside `.claude/command/` include:

- `.claude/command/CLAUDE.md`
- `.claude/command/Lycosa_Claude_Build_Playbook.md`
- `.claude/command/Lycosa_Claude_Build_Playbook_v0.2.0.md`
- `.claude/command/Lycosa_Claude_Build_Playbook_v0.3.0.md`
- AI playbooks
- Agent prompts
- Planning documents

These documents are developer guidance and are part of the repository to support development and integration workflows.

---

## Git Ignore Requirements

Only the `.claude/command/` folder is permitted to contain Claude/AI build documentation in Git. Other root-level Claude files or local-only configurations (such as `settings.local.json`) must be gitignored.

Example:

```gitignore
# Ignore root-level Claude files if any are generated by mistake
/CLAUDE.md
/CLAUDE.local.md
/Lycosa_Claude_Build_Playbook*.md
/*.claude.md

# Allow the .claude/command folder
!.claude/command/
```

Do not commit or stage Claude files outside of `.claude/command/`.

---

## Claude Behavior

Before staging or committing any files, Claude must verify that:

1. All Claude-related playbooks and build documentation are placed within the `.claude/command/` folder.
2. No Claude-specific documentation files (like `CLAUDE.md` or `Lycosa_Claude_Build_Playbook_v0.3.0.md`) are located in the root directory or staged outside of `.claude/command/`.

---

## Repository Rule

The public repository contains application source code, and developer/agent guidance is consolidated in the `.claude/command/` folder.
